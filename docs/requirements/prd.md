# PRD

## 产品名称
人员资料填报助手

## 一句话定义
先从当前电脑里找出某人的相关资料并保留来源，再在用户打开待填文件时提供基于已扫描资料的预览、检索和问答辅助。

## 产品定位
这不是一个“通用网页自动化工具”，而是一个围绕“人员资料类填报”设计的本地资料检索与填表辅助工作台。

MVP 只解决 2 件事：
1. 从当前电脑中找出某人相关信息，并保留来源；信息有冲突时，保留来源并暴露冲突。
2. 用户打开某个要填的表单文件时，系统基于当前文件中已填内容或用户在右侧栏的主动提问，查找之前扫描出的人员信息，并由大模型回答。

## 背景与问题
目标用户经常需要填写以人员资料为核心的申报、登记、审批类表单。现有流程通常是：

1. 在文件夹、聊天记录、Excel、微信附件里反复找资料。
2. 手动判断哪些内容是最新的、可信的。
3. 再把信息逐项搬运到表单里。

当前痛点：
- 信息按文件散落，不按人组织。
- 同一字段可能在多个来源中不一致。
- 每次填报都重复搜集和核对，效率低、错误率高。

## 目标用户
### 核心用户
- 经常填写自己申报资料的个人用户。
- 代他人整理并填写资料的工作人员。
- 需要先汇总多人信息，再统一填报的组织者。

### 非目标用户
- 需要复杂流程编排的 HR SaaS 团队。
- 需要多人在线协同、审批流、权限体系的大型组织。
- 需要支持任意网页自动化操作的 RPA 用户。

## 用户故事映射
### 用户故事 1：填写自己的申报信息
作为经常要提交人员资料类表单的用户，我希望系统根据要求自动整理并填写我的信息，以便减少重复输入，提高填表效率。

对应能力：
- 扫描个人资料来源。
- 生成个人资料卡。
- 在填写文件时按问题查找相关资料。

### 用户故事 2：填写别人的申报信息
作为需要代他人填写人员资料类表单的用户，我希望系统能从已有资料中提取对方信息，以便更快完成代填工作。

对应能力：
- 通过姓名、手机号、证件号等线索定位目标人。
- 汇总该人的多来源资料。
- 处理冲突字段。

### 用户故事 3：用 Excel 汇总多人信息
作为需要统一收集资料的组织者，我希望能够从已有资料中汇总别人的信息，以便集中管理并支持后续填报。

对应能力：
- 支持多人扫描。
- 支持从 Excel 读取多人记录。
- 输出按人组织的资料卡列表。

### 用户故事 4：从微信聊天记录中提取信息
作为资料分散在微信中的用户，我希望系统利用导出的聊天记录和下载文件夹中的附件提取信息，以便把零散内容纳入统一资料库。

对应能力：
- 读取微信导出聊天记录。
- 关联聊天记录和附件文件。
- 将聊天中的有效字段纳入资料卡，并保留来源。

说明：
- 该用户故事保留为后续版本方向，不纳入当前 MVP。

## 产品目标
### MVP 目标
- 用户可以指定一个资料来源范围并发起扫描。
- 系统会在项目下创建 `workspace/` 目录，用于保存扫描产物。
- 系统可以按人聚合资料，而不是按文件展示结果。
- 系统可以生成可追溯的结构化人员资料卡。
- 系统可以显式标记冲突字段和缺失字段。
- 用户打开待填文件后，系统可以读取当前文件中的已填内容作为检索上下文。
- 用户可以在右侧栏提问，系统基于已扫描资料给出回答，并带上来源。

### 成功标准
- 单个用户能在 5 分钟内找到某个人的完整资料卡。
- 用户能在资料卡中看到每个字段的来源文件。
- 用户能区分“已确认”“冲突待确认”“缺失待补充”的字段状态。
- 对未变化文件再次扫描时，系统能跳过重复处理。
- 用户打开待填文件后，能在右侧栏快速看到相关人员信息或得到带来源的回答。

## 版本范围
### MVP 范围
支持输入来源：
- 用户手动指定的本地文件夹。

支持扫描文件类型：
- `doc`
- `docx`
- `xlsx`
- `xls`
- `csv`
- `md`
- `txt`

支持输出结果：
- 人员列表。
- 单人资料卡。
- 冲突核对视图。
- 来源追溯视图。
- 待填文件预览视图。
- 右侧问答栏。
- `workspace/` 中的人员 Markdown 文档。
- `workspace/` 中由 `doc` 转换得到的 `docx` 文件。

### 暂不支持
- PDF/OCR 高精度识别承诺。
- `Pages` 等小众格式深度支持。
- 微信聊天记录导出文件作为首版正式输入源。
- 自动打开浏览器并提交网页表单。
- 自动生成 Word/Excel 成品申报文件。
- 云端账号体系。

## 核心流程
### 流程 1：配置扫描源
1. 用户选择一个本地目录作为扫描源。
2. 用户选择扫描模式：
   - 首次全量扫描
   - 增量扫描
3. 系统检查并创建 `workspace/` 目录。

输出：
- 扫描任务创建成功。

### 流程 2：扫描与抽取
1. 系统遍历扫描源中的文件。
2. 命中 `doc` 文件时，先将其转换为 `docx`，并把转换后的文件写入 `workspace/converted/`。
3. 根据文件类型选择解析策略。
4. 提取候选人员字段和来源元数据。
5. 为每个来源生成抽取结果。

输出：
- 文件级抽取结果。
- 文件扫描状态。

### 流程 3：按人聚合
1. 系统根据身份线索将来源结果归并到“人”。
2. 对同一字段的多值做合并与冲突判断。
3. 生成人员资料卡草稿。

输出：
- 人员列表。
- 每人的字段状态汇总。

### 流程 4：人工核对
1. 用户打开某个人的资料卡。
2. 查看冲突字段候选值及其来源。
3. 选择最终值，或标记为待补充。
4. 系统记录用户确认结果。

输出：
- 已确认字段。
- 待补充字段。

### 流程 5：打开待填文件
1. 用户打开一个待填写的本地文件，文件类型支持 `doc`、`docx`、`xlsx`、`xls`、`csv`、`md`、`txt`。
2. 系统解析当前文件中已经填写的内容，提取候选人线索、字段线索和上下文。
3. 右侧栏默认展示与当前文件最相关的人员资料、来源和冲突提示。

输出：
- 文件预览结果。
- 当前文件关联的候选人员和字段线索。

### 流程 6：右侧栏检索与问答
1. 用户在右侧栏直接提问，例如“这个人的毕业院校是什么”“手机号从哪个文件来的”。
2. 系统基于当前打开文件的上下文和已扫描资料检索相关人员信息。
3. 大模型生成回答，并明确引用来源；若信息冲突，优先暴露冲突而不是强行给单一结论。
4. 用户可继续追问，直到找到可用于填写的资料。

输出：
- 带来源的回答。
- 冲突提示。

## 功能需求
### FR-1 扫描源管理
系统必须支持用户配置本地扫描源，并记录最近一次扫描配置。

要求：
- 支持用户指定一个本地文件夹作为扫描输入。
- 支持手动触发重新扫描。
- 支持显示最近扫描时间。
- 首次使用时自动创建 `workspace/` 目录。
- `workspace/` 至少包含人员资料文件和 `doc -> docx` 转换产物。

验收标准：
- 用户可以在不重新配置全部路径的情况下再次扫描。
- 扫描完成后，`workspace/` 中可以看到生成的资料文件和转换后的 `docx` 文件。

### FR-2 文件扫描与增量更新
系统必须支持全量扫描和增量扫描。

要求：
- 使用 SQLite 记录扫描状态。
- 记录文件路径、文件类型、最后修改时间、文件哈希或等效指纹。
- 对未变化文件跳过重复抽取。
- 对新增、变化、删除文件更新索引状态。

验收标准：
- 第二次扫描同一目录时，未变化文件不重复进入抽取流程。
- 系统可从 SQLite 中查到某个文件是否扫描过、上次扫描时间是什么。

### FR-3 文件解析与抽取
系统必须根据文件类型选择解析方式。

MVP 支持优先级：
- `doc`
- `docx`
- `xlsx` / `xls`
- `csv`
- `md` / `txt`

要求：
- `doc` 文件必须先转换为 `docx`，再进入统一解析流程。
- 转换后的 `docx` 文件必须写入 `workspace/converted/`。
- SQLite 需要记录原始 `doc` 与转换后 `docx` 的对应关系。
- 每份文件都要生成抽取状态：成功、部分成功、失败、跳过。
- 失败文件不能静默消失，必须在来源列表中可见。

验收标准：
- 用户能看到哪些文件被解析，哪些没有解析成功。
- 用户能根据原始 `doc` 文件定位到对应的转换后 `docx` 文件。

### FR-4 以人为中心的聚合
系统必须按人组织结果，而不是按文件组织结果。

聚合优先线索：
1. 证件号
2. 手机号
3. 邮箱
4. 姓名 + 单位/学校

要求：
- 高置信度线索自动合并。
- 低置信度线索不自动合并，保留为独立候选人。
- 支持后续人工合并。

验收标准：
- 多个来源命中同一人的情况下，用户在列表中只看到一个人条目。

### FR-5 人员资料字段模型
系统必须统一输出结构化字段。

MVP 字段分组：
- 基础信息：姓名、性别、手机号、邮箱、证件号、出生日期、地址
- 教育信息：最高学历、学位、毕业院校、专业、毕业时间
- 职业信息：单位、部门、岗位、职称、职称取得时间
- 其他信息：证书、荣誉、备注

字段对象至少包含：
- 字段名
- 字段值
- 字段状态
- 来源列表
- 最近确认时间
- 是否人工确认

字段状态定义：
- `confirmed`：已确认，可用于填报
- `conflict`：存在多个候选值，待人工确认
- `missing`：当前无可靠值
- `candidate`：已抽取但未确认

验收标准：
- 每个展示字段都能落入固定 schema，不允许自由文本散落在主资料区。

### FR-6 来源追溯
系统必须保留每条字段的来源。

来源记录至少包含：
- 来源文件名
- 来源文件路径
- 文件类型
- 抽取时间
- 原文片段或定位信息

验收标准：
- 用户点击任一字段时，可以看到至少一个来源记录。

### FR-7 冲突识别与处理
系统必须识别同一字段的多值冲突，并支持人工选择最终值。

冲突定义：
- 同一人、同一字段下存在两个及以上非空且不相等的候选值。

要求：
- 冲突字段在列表和详情中显式高亮。
- 用户可从候选值中选一个作为最终值。
- 用户可暂不选择，字段保持 `conflict` 状态。

验收标准：
- 用户处理冲突后，资料卡与待填字段结果同步更新。

### FR-8 人员资料卡输出
系统必须为每个人生成一份资料卡，并支持写入 Markdown。

要求：
- 每人一个 Markdown 文件，写入 `workspace/people/`。
- 文件结构固定为：基础信息、教育信息、职业信息、其他、冲突记录、来源列表。
- 不修改原始资料文件。
- 每次保存人员 Markdown 文件时，系统都要重新计算并更新该人员的冲突状态。
- SQLite 需要记录存在冲突的人员 Markdown 文件及其当前冲突状态。

验收标准：
- 用户可直接打开某个人的 Markdown 文档查看完整资料。
- 某人的冲突被解决后，再次保存对应 Markdown 文件，SQLite 中的冲突状态会同步变化。

### FR-9 待填文件预览
系统必须支持用户打开待填文件，并基于文件当前内容建立检索上下文。

MVP 支持打开的文件类型：
- `doc`
- `docx`
- `xlsx`
- `xls`
- `csv`
- `md`
- `txt`

要求：
- 系统要能读取当前文件中已填写的可解析内容。
- 系统要从当前文件里提取候选人线索、单位、学校、手机号、证件号等上下文。
- 右侧栏默认展示与当前文件最相关的人员资料摘要、来源和冲突状态。
- 若当前文件无法可靠识别目标人，必须明确提示“不确定”，而不是隐式绑定到某个人。

验收标准：
- 用户打开一个待填文件后，不需要重新输入完整姓名，也能看到相关候选资料或明确的不确定提示。

### FR-10 右侧栏问答
系统必须支持用户在右侧栏主动提问，并由大模型基于已扫描资料作答。

要求：
- 问答必须结合当前打开文件的内容和历史扫描结果。
- 回答必须带来源；若是模型推断，必须显式区分“来源事实”和“推断结论”。
- 当信息存在冲突时，回答必须先暴露冲突值及其来源。
- 用户可以继续追问来源、冲突原因和缺失项。

验收标准：
- 用户提问“这个人的毕业院校是什么”时，系统返回答案或冲突说明，并附至少一个来源。
- 用户提问“手机号从哪个文件来的”时，系统能返回对应来源文件。

### FR-11 扫描日志与可见性
系统必须提供最小可见性，避免用户不知道系统做了什么。

要求：
- 展示扫描开始时间、结束时间、处理文件数、跳过文件数、失败文件数。
- 展示按人的聚合结果数量。

验收标准：
- 扫描完成后，用户能理解本次处理范围和结果。

## 核心界面
### 页面结构总览
MVP 建议采用 4 个一级页面 / 工作区：
1. 扫描入口页
2. 人员列表页
3. 人员工作台
4. 待填文件辅助页

其中“右侧问答栏”不是独立页面，而是“待填文件辅助页”中的固定侧栏区域。

### 1. 扫描入口页
页面目标：
- 让用户完成扫描源配置、发起扫描，并理解本次扫描做了什么。

进入方式：
- 首次启动默认进入。
- 也可从全局导航中进入“扫描”。

页面内容：
- 扫描源选择区：展示当前已选择的本地目录。
- 扫描模式区：支持首次全量扫描、增量扫描。
- 扫描结果摘要区：展示最近一次扫描时间、处理文件数、跳过文件数、失败文件数、聚合后人数。
- 最近任务区：展示最近一次任务状态和异常提示。

关键操作：
- 选择或更换扫描目录。
- 发起全量扫描。
- 发起增量扫描。
- 查看失败文件和失败原因。

状态要求：
- 首次未配置时，页面主按钮应引导用户先选择目录。
- 扫描进行中时，需展示明确的进度状态，避免用户误以为系统无响应。
- 扫描完成后，页面应给出可进入“人员列表页”的下一步入口。
- 若 `workspace/` 初始化失败，页面必须明确报错，不可静默失败。

### 2. 人员列表页
页面目标：
- 让用户以“人”为单位快速找到目标对象，而不是回到文件级逐个翻看。

进入方式：
- 扫描完成后自动推荐进入。
- 从全局导航中进入“人员”。

页面内容：
- 列表工具区：支持关键词搜索、状态筛选、排序。
- 人员列表区：每行至少展示姓名、资料完整度、冲突数、最近更新时间、主要身份线索摘要。
- 快速预览区或行内摘要：展示该人的基础信息摘要和高风险提醒。

关键操作：
- 按姓名、手机号、证件号、单位等线索搜索。
- 按“有冲突”“待补充”“最近更新”筛选。
- 点击某个人进入人员工作台。

状态要求：
- 无扫描结果时，页面要明确提示先去扫描入口页建立资料库。
- 搜索无结果时，要提示可调整关键词，而不是展示空白页。
- 冲突人数较多时，默认应支持按冲突优先排序，帮助用户先处理高风险对象。

### 3. 人员工作台
页面目标：
- 让用户围绕单个人完成查看、核对、确认和追溯。

进入方式：
- 从人员列表页点击进入。

页面结构：
- 顶部基础信息区：展示姓名、别名、主要身份线索、更新时间、当前资料状态。
- 主内容区采用 3 个视图切换：
  - 资料概览
  - 冲突核对
  - 来源文件

#### 3.1 资料概览
页面目标：
- 让用户快速看到该人的结构化资料现状，判断哪些字段可直接用于填报。

页面内容：
- 基础信息、教育信息、职业信息、其他信息四个分组。
- 每个字段展示字段值、字段状态、是否人工确认、最近确认时间。
- 对 `missing`、`candidate`、`conflict` 字段做显式区分。

关键操作：
- 点击字段查看来源。
- 对可确认字段执行“确认使用”。
- 对缺失字段标记“待补充”。

状态要求：
- `confirmed` 字段应是默认最稳定、最适合填报的展示状态。
- `conflict` 字段不能被视觉上伪装成已确认值。

#### 3.2 冲突核对
页面目标：
- 让用户集中处理冲突字段，而不是在资料概览里逐项翻找。

页面内容：
- 冲突字段列表，按字段分组展示多个候选值。
- 每个候选值必须展示来源、抽取时间、原文片段或定位信息。
- 当前已选最终值与未决状态需清晰区分。

关键操作：
- 从候选值中选择最终值。
- 保持未决，稍后处理。
- 跳转查看对应来源文件。

状态要求：
- 所有冲突字段都必须可见，不允许只显示“有冲突”而不给具体候选值。
- 用户完成选择后，资料概览中的字段状态要立即同步。

#### 3.3 来源文件
页面目标：
- 让用户确认某条信息究竟来自哪里，建立对系统结果的信任。

页面内容：
- 来源文件列表，展示文件名、路径、类型、抽取状态、最近抽取时间。
- 点击来源文件后，可查看该文件贡献过哪些字段。
- 若有失败文件，也应在该视图中可见。

关键操作：
- 按文件类型或抽取状态筛选来源。
- 从字段反查来源文件。
- 定位到转换后的 `docx` 文件。

状态要求：
- 失败文件必须展示失败状态，不能从来源链路中消失。
- 若字段来自多个文件，该视图应支持完整展示全部来源，而不是只保留一个“主来源”。

### 4. 待填文件辅助页
页面目标：
- 在用户实际填写文件时，基于当前文件内容提供相关人员资料、来源和问答辅助。

进入方式：
- 用户主动打开某个待填文件时进入。

页面布局：
- 左侧为待填文件预览区。
- 中间为候选人员与资料摘要区。
- 右侧为问答栏。

#### 4.1 待填文件预览区
页面内容：
- 当前打开文件内容的可解析预览。
- 从文件中提取出的候选人线索、单位、学校、手机号、证件号等上下文。
- 当前文件类型与解析状态提示。

关键操作：
- 查看当前文件中已识别出的线索。
- 在多个候选人之间切换查看匹配结果。

状态要求：
- 若文件可以打开但解析不完整，要明确提示“部分可解析”。
- 若无法可靠识别目标人，页面必须显示“不确定”，并给出候选人而不是默认绑定。

#### 4.2 候选人员与资料摘要区
页面内容：
- 当前最相关候选人的基础摘要。
- 关键字段状态，尤其是可直接填报项、冲突项、缺失项。
- 与当前文件最相关的来源提示。

关键操作：
- 在候选人之间切换。
- 一键跳转到对应人员工作台。

状态要求：
- 默认排序应优先展示与当前文件上下文匹配度最高的人。
- 若无任何可匹配人员，必须明确提示“未找到可关联资料”，而不是展示空摘要卡。

#### 4.3 右侧问答栏
页面目标：
- 让用户围绕当前待填文件持续提问，并获得带来源的答案。

页面内容：
- 默认推荐问题或资料摘要。
- 用户输入框。
- 多轮回答记录。
- 每条回答对应的来源和冲突提示。

关键操作：
- 提问当前字段相关问题。
- 追问来源、冲突原因、缺失项。
- 从回答中跳转到来源文件或人员工作台。

状态要求：
- 回答中必须区分“来源事实”和“模型推断”。
- 若答案依赖冲突字段，优先展示冲突，而不是给出伪确定答案。
- 若当前上下文不足以回答，系统应明确说明缺少什么信息。

## 数据与存储要求
### 工作目录
系统在运行时创建 `workspace/` 目录，作为本地工作区。

建议结构：
- `workspace/people/`：按人生成的 Markdown 资料卡。
- `workspace/converted/`：由 `doc` 转换得到的 `docx` 文件。
- `workspace/app.db`：SQLite 数据库文件。

### 存储引擎
MVP 使用 SQLite 作为本地唯一结构化存储。

SQLite 至少负责存储：
1. 哪些文件已经扫描过，以及扫描时间、文件状态和文件指纹。
2. 原始 `doc` 文件与转换后 `docx` 文件的对应关系。
3. 存在冲突的人员 Markdown 文件路径及其当前冲突状态。

## 非功能要求
### 可追溯性
- 任何进入资料卡的字段都必须能回到来源。

### 安全性
- 默认只读原始资料，不覆盖、不改写。
- 本地优先，不要求上传云端。
- `doc` 转换后的 `docx` 只写入 `workspace/converted/`，不覆盖原始 `doc`。

### 性能
- 100 个文件以内的首次扫描应在可接受等待时间内完成。
- 增量扫描应明显快于首次全量扫描。

### 可扩展性
- 后续可新增文件类型解析器。
- 后续可新增更强的表单自动填充模块，而不重做资料层。

## 边界与风险
### 已知边界
- 微信聊天和附件命名可能不规范，首版只能做到“可挂接、可追溯、部分提取”。
- 图片、扫描件、PDF 的识别质量不稳定，首版不把它们作为核心依赖。
- 人员合并存在误合并风险，低置信度必须人工介入。

### 产品风险
- 如果当前打开文件的上下文识别不准，右侧问答容易关联错人。
- 如果不保留来源，用户不会信任自动抽取结果。
- 如果回答不区分“来源事实”和“模型推断”，用户会误把推断当作已确认信息。

## 验收标准
### 场景 1：填写自己的申报信息
- 给定一个包含个人简历、身份证扫描件、Excel 联系方式的目录，系统可生成单人资料卡。
- 用户可看到基础信息、教育信息、职业信息。

### 场景 2：填写别人的申报信息
- 给定多个候选人的混合资料，用户可通过人员列表定位目标人。
- 目标人的冲突字段会被显式标记。

### 场景 3：用 Excel 汇总多人信息
- 给定一份多人 `xlsx`、`xls` 或 `csv`，系统可生成多人资料卡，而不是只输出一份文件级结果。

### 场景 4：`doc` 文件转换
- 给定一个包含 `doc` 文件的目录，系统会先把 `doc` 转换为 `docx`，并将转换产物写入 `workspace/converted/`。
- SQLite 中可查到原始 `doc` 和转换后 `docx` 的对应关系。

### 场景 5：打开待填文件辅助查找
- 用户打开一个待填的 `docx`、`xlsx`、`md` 或 `txt` 文件后，系统能基于当前已填内容识别候选人线索。
- 右侧栏会展示相关资料摘要、来源和冲突提示。
- 用户可继续提问，系统返回带来源的答案。

### 场景 6：重复扫描
- 对未变化目录再次扫描时，系统会跳过未变化文件，并保留既有资料卡。
- SQLite 中会更新本次扫描时间，并复用历史扫描记录。

## 一句话结论
首版产品应该先把“按人找资料、保留来源、暴露冲突、打开待填文件后的侧栏检索与问答”做扎实，再考虑更强的自动填充能力。
