SQLite 表设计可视化

这页把 `profile-fill-assistant` 的核心 SQLite 表拆成 3 个工作域：扫描输入、按人聚合、待填文件会话。 桌面端会直接把关联线叠在表卡片之间，便于快速看清主键、外键和 1:1 / 1:N 关系。

9 核心表
9 已标出的主关联
3 领域分组
2 关系类型：1:1 / 1:N
扫描与来源链路 按人聚合与字段结果 待填文件会话 实线 = 1:N 虚线 = 1:1

Ingest

扫描任务与原始文件

scan_jobs

记录每次全量或增量扫描任务的元信息。

任务主表
  • PKscan_job_id任务主键
  • COLroot_path扫描根目录
  • COLscan_modefull / incremental
  • COLstatusrunning / success / failed

source_records

文件级主表，记录扫描、解析和抽取状态。

来源主表
  • PKsource_id来源主键
  • FKscan_job_id关联 scan_jobs
  • COLfile_path原始文件绝对路径
  • COLextract_statussuccess / failed / skipped

conversion_records

记录 `doc -> docx` 转换映射。

转换映射
  • PKconversion_id转换主键
  • FKsource_id关联 source_records
  • COLsource_doc_path原始 doc 路径
  • COLconverted_docx_path转换后 docx 路径

Profiles

按人聚合、字段结果、冲突摘要

person_profiles

按人聚合后的主实体，服务人员列表与工作台。

人员主表
  • PKperson_id人员主键
  • COLdisplay_name默认展示姓名
  • COLidentity_keys_json证件号、手机号、邮箱等身份线索
  • COLconflict_count当前冲突字段数

field_entries

当前最终字段结果，供资料概览和默认摘要使用。

字段当前值
  • PKfield_entry_id字段记录主键
  • FKperson_id关联 person_profiles
  • COLfield_key字段名，例如 phone
  • COLcanonical_value当前最终值

field_candidates

字段候选值明细，用于冲突核对和来源追溯。

候选明细
  • PKcandidate_id候选值主键
  • FKfield_entry_id关联 field_entries
  • FKsource_id关联 source_records
  • COLcandidate_value候选字段值

conflict_records

人员级和字段级冲突摘要，便于快速筛选和处理。

冲突摘要
  • PKconflict_id冲突主键
  • FKperson_id关联 person_profiles
  • FKfield_entry_id关联 field_entries
  • COLresolution_statusopen / resolved / ignored

Sessions

待填文件上下文与候选人匹配

form_sessions

记录一次待填文件打开会话和上下文快照。

会话主表
  • PKsession_id会话主键
  • COLopened_file_path当前打开文件路径
  • COLcontext_entities_json从文件中识别出的上下文
  • FKmatched_person_id默认匹配人员，可为空

form_session_candidates

保存某次会话下的候选人排序结果。

匹配结果
  • PKsession_candidate_id候选记录主键
  • FKsession_id关联 form_sessions
  • FKperson_id关联 person_profiles
  • COLmatch_score候选匹配得分

关系清单

scan_jobs → source_records `scan_jobs.scan_job_id` 通过 `source_records.scan_job_id` 建立 1:N 关系。
source_records → conversion_records `source_records.source_id` 对应 `conversion_records.source_id`，关系是 1:1。
person_profiles → field_entries 一个人拥有多个当前字段结果，因此是 1:N。
field_entries → field_candidates 每个字段条目可以对应多个候选值，关系是 1:N。
source_records → field_candidates 候选值保留来源文件，因此 `source_id` 指回 `source_records`。
person_profiles → conflict_records 一个人可以存在多个冲突摘要，关系是 1:N。
field_entries → conflict_records 每个冲突记录也会落到具体字段条目上，关系是 1:N。
form_sessions → form_session_candidates 一个待填文件会话会产出多个候选人排序结果，关系是 1:N。
person_profiles → form_session_candidates 候选人记录里的 `person_id` 指向人员主表，形成多对一反向视角。

注：`form_sessions.matched_person_id` 也会指向 `person_profiles.person_id`，但由于它是“当前默认匹配”而不是候选列表主链路，这里以辅助关系处理。
